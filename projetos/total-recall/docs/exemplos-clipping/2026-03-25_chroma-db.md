## O que isso muda no projeto como um todo

Esta é a parte relevante para o macro:

**O Hot Store (LangGraph SqliteSaver) e o Warm Store (memória semântica) podem viver no mesmo arquivo SQLite** — ou em arquivos distintos, mas com a mesma tecnologia, sem infraestrutura nova.

```
novex-memory.**`db`**
  ├── briefing_threads        ← Hot Store (LangGraph state)
  ├── score_calibrations      ← Hot Store
  ├── memory_patterns         ← Hot Store  
  ├── chunks_vec              ← Warm Store (vetores)
  ├── chunks_fts (FTS5)       ← Warm Store (keyword)
  └── embedding_cache         ← Warm Store (cache)
```

Para o sistema de 7+1 agentes isso significa:
- **Nenhuma infraestrutura nova** — o SQLite já existe por causa do LangGraph
- **Todos os agentes acessam o mesmo store** sem broker intermediário
- **Novos agentes via MIP** plugam no mesmo **`DB`** — sem configurar um **`ChromaDB`** separado
- **Embedding provider configurável** — começa sem embeddings (FTS5 gratuito), adiciona OpenAI `text-embedding-3-small` ou local quando quiser
- **ADR-001 cai por terra** — **`ChromaDB`** e Pinecone saem do plano; SQLite vai até produção (é exatamente o que OpenClaw valida em escala)
